# TribeOS Design System

**Status:** Approved v1.0

---

## Purpose

This document defines the visual language, UI standards, and interaction principles for TribeOS. It is the single source of truth for frontend implementation.

Every UI generated by Cursor or implemented manually must follow this document. Do not invent colors, spacing, typography, or component styles outside this specification. For the semantic token names used in code, see [`design_tokens.md`](./design_tokens.md).

---

## Design Philosophy

TribeOS is not a generic admin dashboard. It is an internal operating system for a premium event management company.

The interface should feel: **premium, minimal, modern, confident, professional, fast, clean.**

Prioritize clarity over decoration. Avoid unnecessary gradients, glassmorphism, or flashy animations.

## Inputs

- Filled dark background, subtle border, rounded.
- Focus ring is orange.
- Labels always visible — never rely on placeholders alone.

---

## Tables

- Dark background, sticky headers, alternating row hover.
- Right-align numbers; left-align text.
- Status badges.
- Pagination, search, sorting, filtering always supported.

---

## Cards

- Rounded (14px radius), small shadow, consistent padding (24px).
- Structure: Title, Subtitle, Content, Actions, optional Footer.

---

## Forms

- Always use React Hook Form + Zod.
- Validation shown below fields; required fields marked.
- Never use browser alerts.

---

## Dialogs

- Centered, rounded, maximum width 720px.
- Background blur; ESC closes; click-outside configurable.

---

## Navigation

- Left sidebar + top header.
- Breadcrumbs optional.
- Current page highlighted; icons on every navigation item.

---

## Empty States

Every screen should have an empty state. Include:

- Illustration (optional)
- Title
- Description
- Primary action

Never show blank tables.

---

## Loading States

- Use Skeleton components.
- Avoid spinners unless waiting longer than 500ms.

---

## Error States

- Friendly language; explain what happened.
- Provide a recovery action.
- Never expose stack traces.

---

## Badges

Use for: Status, Priority, Invoice State, Payment State, Vendor State. Use consistent colors.

---

## Notifications

Use Toasts, positioned **top right**.

| Type | Color |
| --- | --- |
| Success | Green |
| Warning | Orange |
| Error | Red |
| Info | Blue |

---

## Charts

- **Library:** Recharts
- Minimal grid, readable labels, responsive, dark theme, consistent colors.

---

## Animations

- Short: 150–200ms, ease-out.
- Avoid bouncing. No page-transition animations. Micro-interactions only.

---

## Accessibility

- Minimum WCAG AA contrast.
- Keyboard accessible; visible focus states.
- ARIA labels where required; screen-reader friendly.

## Tailwind Rules

- Use semantic utility classes.
- Avoid inline styles.
- Extract repeated patterns into reusable components.
- Prefer semantic token classes (`bg-primary`, `text-muted`, `border-border`) over raw hex — see [`design_tokens.md`](./design_tokens.md).

---

## Design Principles

Every screen should answer:

- Where am I?
- What can I do?
- What needs attention?
- What changed?
- Can I complete my task quickly?

---

## UX Principles

- Minimize clicks.
- Avoid modal overload.
- Prefer inline editing.
- Preserve user context.
- Autosave where appropriate.
- Confirm destructive actions.
- Never interrupt the user unnecessarily.

---

## Financial Screens

- Financial data must be highly readable.
- Right-align all numeric values; use tabular numbers.
- Highlight totals; show currency consistently.
- Negative values use the danger color; positive values use the success color.

---

## Event Management Screens

- Prioritize timeline visibility.
- Show vendor status clearly.
- Display budget utilization.
- Surface overdue actions; highlight blockers.
